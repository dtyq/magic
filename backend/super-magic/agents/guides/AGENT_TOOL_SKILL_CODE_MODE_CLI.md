# Agent 能力暴露范式：工具、Skill、Code Mode 与 CLI

本文梳理三种 Agent 能力暴露方式的机制、适用场景与取舍依据，并说明 Code Mode 的独立角色，供设计新能力时参考。

## 两种主流范式

当前面向 Agent 提供能力主要有两种方式：**工具（Tool Calling）** 和 **Skill + Code Mode**。两种方式分别对应不同的能力暴露策略，也代表了当前较新的主流实践：

- 高频、原子、边界清晰的能力，继续通过工具直接挂载给 Agent
- 低频、复杂、带工作流性质的能力，通过 Skill 按需加载说明，并由 Code Mode 承担实际执行

CLI 是第三种形式，本质上是 Skill 的一种执行方式的变体，其核心价值在分发侧而非使用侧，后文单独说明。

---

## 三种能力暴露方式

### 工具（Tool Calling）

把能力直接注册成 LLM 可调用工具。模型推理时看到工具名、参数 schema 和 description，然后发起结构化调用。每次工具调用的结果都会原样流回模型上下文。

**适合**：高频、原子、边界清晰的能力；参数化明确、一次调用即可完成的操作。

**代价**：

- **工具定义占用上下文**：工具数量越多，每次推理前需要读取的定义越多，成本线性上升。接入数十个 MCP server、成百上千个工具时，光工具定义就可能消耗数十万 token。
- **中间结果流经上下文**：每次工具调用的返回值都必须通过模型。如果连续调用多个工具，所有中间数据都会在上下文中累积。一次多步的数据处理任务，可能仅凭中间结果就消耗数万 token。

---

### Skill

把能力打包成"按需加载的任务知识"。Skill 本质上是一组提示词，告诉模型：什么时候用这套能力、整体步骤是什么、有哪些判断依据、有哪些示例。

Agent 平时不需要背着这套完整说明，只有在识别到某类任务时才去读取对应 Skill。Skill 里的执行方式没有规定——可以是 Code Mode、CLI 命令、脚本，也可以只是自然语言步骤指引。

**Skill 解决的是按需加载问题，不规定执行方式。**

Skill 最常见的搭配是 Code Mode：Skill 告诉模型"这件事该怎么做"，Code Mode 负责执行——模型读完 Skill 后写代码，代码调用底层工具完成工作。两者分工明确：Skill 解决"复杂工作流如何按需进入上下文"，Code Mode 解决"复杂工作流如何执行"。两者概念上相互独立，但配合使用是当前的主流实践。

**适合**：低频、复杂、步骤多、需要判断和编排的能力；不希望常驻上下文的工作流。

---

### CLI

将能力封装成命令行工具，Agent 通过写 shell 命令完成任务，并可通过 `某某CLI --help` 动态发现参数和用法。

**优势**：分发效率高，天然保密——发布二进制即可分发，无需暴露源码。对有技术壁垒的商业工具是合理的分发形式。

**局限**：本质上是让 Agent 写 shell。shell 的训练语料远少于代码，模型处理复杂逻辑时能力明显弱于写 Python。CLI 在 Agent 侧的优势主要体现在简单高频命令，而非复杂工作流。

对于简单的任务（如 Markdown 转 PDF），CLI 省 token，但在 Code Mode 下封装一行 Python 调用效果相当。CLI 的核心价值在**分发侧**，不在**执行侧**。

---

## 设计原则：入口唯一

**同一个 Agent-facing 能力只保留一个主入口。**

不要把同一个能力既做成直接挂载工具，又做成 Skill 版本。一旦同时暴露两套入口，模型面临歧义，开发者也面临双份说明和维护负担。

需要区分"Agent-facing 入口"和"运行时实现"：选择了 Skill 的能力，底层完全可以调用工具——Skill 指引往往正是通过调工具完成工作的。真正不应该并存的，是同一能力在 Agent 面前出现两套平行使用入口。

**判断标准**：

- 高频、稳定、短路径、参数化明确 → 挂载成工具
- 低频、复杂、步骤多、需要判断和编排 → 做成 Skill

---

## Code Mode 是什么

Code Mode 是一种执行机制，不是能力暴露方式，也不是 Skill 的专属配套。

它的含义是：模型不直接发起结构化工具调用，而是先生成 Python 代码，再由代码去调用底层工具或 MCP 原语。Cloudflare 率先用"Code Mode"命名了这种模式，Anthropic 在 2025 年 11 月的工程博客中引用并正式认可了这一概念。

```
模型生成 Python 代码
    │
    ▼
执行环境运行临时脚本
（中间结果在执行环境中处理，不流经模型上下文）
    │
    ▼
sdk.tool.call(...) 或 sdk.mcp.call(...)
    │
    ▼
服务端路由到当前 AgentContext，完成真实工具调用
```

Code Mode 常见于 Skill 场景——Skill 指引模型"写代码来完成这件事"——但它本身是独立的执行路径，不依赖 Skill 单独存在。

---

## 为什么 Code Mode 比直接工具调用更高效

Code Mode 的效率优势来自两个方面。

### 中间结果不流经模型

直接工具调用中，每次调用的返回值都必须通过模型：模型看到结果 → 决定下一步 → 发起下一次调用。每一步都会把中间数据写进上下文。

Code Mode 中，模型只写一段代码，代码在执行环境里跑完整个流程，只把最终结果（或模型明确要求打印的内容）返回给模型。所有中间数据在执行环境内部流转，不进入上下文。

Anthropic 的实测数据：将工具 API 以代码方式暴露，而非直接工具调用，可将 token 消耗从 150,000 减少到 2,000——节省 **98.7%**。

### 复杂逻辑在代码里处理，不用多轮推理

循环、条件判断、异常处理、数据过滤——这些逻辑用代码写出来一步完成，不需要多次返回模型等待决策。以 Anthropic 的例子为例：等待 Slack 部署通知这件事，用工具调用需要模型反复发起 sleep + 查询，每次都是一次完整的推理；用代码写一个 while 循环，执行环境处理等待，模型只需推理一次。

---

## 为什么模型更擅长写代码

Code Mode 能生效，根本原因是：**大模型更擅长写代码，而不是发起结构化工具调用**。

互联网上代码的训练语料极其丰富——开源项目、技术文档、技术问答构成了模型训练数据的重要组成。相比之下，结构化工具调用是相对新颖的范式，模型见过的样本远少于代码。这意味着：

- 让模型"写一段 Python 完成这件事"，泛化能力远强于"选一个工具然后填 JSON 参数"
- 代码可以内联条件判断、循环、数据变换等复杂逻辑；工具调用是单次结构化请求，无法原生表达这些

Anthropic 的 Programmatic Tool Calling 和 HuggingFace 的 smolagents，都在各自产品中选择了以沙箱化 Python 执行作为 Agent 的编排层，而不是纯结构化工具调用。这是当前业界在这个方向上的主流趋势。

---

## Code Mode 的代价

Code Mode 不是免费的。代码执行引入了直接工具调用没有的复杂度：

- **需要安全的执行环境**：沙箱隔离、资源限制（内存、CPU、网络访问范围）、运行时监控
- **执行延迟**：单次简单工具调用约 45ms；代码生成和执行约 380ms，适合多步任务，不适合高频原子操作
- **安全边界更宽**：生成的代码原则上可以访问执行环境内的任何资源，需要明确的沙箱设计

这些成本应在选择执行方式时纳入考量，而不是默认所有能力都迁移到 Code Mode。

**本项目的情况**：Super Magic 专门设计运行在自研沙箱中，沙箱基础设施已由平台统一提供和维护。这意味着 Code Mode 所需的安全执行环境已是既有条件，而不是额外负担。因此在本项目中，Code Mode 的代价已被大幅摊薄，凡是适合用 Skill + Code Mode 的复杂工作流，都应优先考虑这条路径。

---

## 总结：从旧范式到新范式

今天的 Agent 能力设计，正在从"默认一切都挂工具"的旧范式，逐步转向"高频能力保留工具，低频复杂能力转向 Skill + Code Mode"的新范式。工具不再是唯一入口，而是高频原子能力的入口；Skill 也不只是文档，而是低频复杂能力的按需入口；Code Mode 则是这类复杂能力的执行引擎。三者不是完全对立，但在 Agent-facing 设计上必须分层清楚、入口唯一、职责单一。

## CodeModeOnly 单活

当能力只应该通过 Skill + Code Mode 使用，不应该直接挂载为工具时，工具类声明 `code_mode_only = True`。这类工具仍然使用标准 Tool 实现承载执行逻辑和前端展示，但不会出现在 LLM tools 列表中；如果误挂到 agent 的 `tools:`，执行层会拦截。开发者只需要维护对应 Skill 作为唯一 Agent-facing 入口。

适合场景：

- 工具数量多，直接挂载会污染主上下文
- 单次任务需要多步调用、筛选、聚合，适合在 Code Mode 中处理
- 不希望底层渠道、供应商、内部接口在直接工具列表中曝光

## 对比总览

**三种能力暴露方式：**

| | 工具 | Skill（常搭配 Code Mode） | CLI |
|--|--|--|--|
| 能力类型 | 高频原子 | 低频复杂 | 高频简单（外部分发） |
| 上下文占用 | 常驻 | 按需加载 | 按需（shell） |
| 执行方式 | 直接结构化调用 | 任意（Code Mode / 脚本等） | 模型写 shell |
| 源码保密 | 不涉及 | 通常无法保密 | 可以保密 |
| 适合场景 | 核心原语 | 复杂工作流 | 外部商业工具分发 |

**两种执行机制（直接工具调用 vs Code Mode）：**

| | 直接工具调用 | Code Mode |
|--|--|--|
| 单次延迟 | ~45ms | ~380ms |
| 多步任务效率 | 每步都经过模型 | 中间结果留在执行环境 |
| Token 消耗 | 随步骤数线性增长 | 仅最终结果进入上下文 |
| 安全模型 | 工具边界天然隔离 | 依赖沙箱设计 |
| 适合场景 | 高频原子操作 | 多步编排、复杂逻辑 |

**两种主流范式对比：**

| | 工具为主（旧范式） | Skill + Code Mode（新范式） |
|--|--|--|
| 核心逻辑 | 能力即工具，挂载即可用 | 高频挂工具，低频做成 Skill |
| 上下文压力 | 工具越多越重 | 按需加载，主线保持精简 |
| 执行复杂度 | 单步原子操作 | 可编排多步、含分支逻辑 |
| 适合阶段 | 能力少、场景清晰时 | 能力多、工作流复杂时 |
