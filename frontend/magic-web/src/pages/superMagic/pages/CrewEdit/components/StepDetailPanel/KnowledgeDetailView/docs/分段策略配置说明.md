# 分段策略配置数据结构说明

## 一、核心概念

分段策略（fragment_config）用于控制文档如何被切分成小的片段（chunks），以便进行向量化存储和检索。

### 1.1 三种策略模式

| 策略 | mode值 | 说明 | 适用场景 |
|------|--------|------|---------|
| **自动分段** | 2 (AUTO) | 系统自动识别文档结构并智能切分 | 通用文档，不需要精细控制 |
| **层级分段** | 3 (HIERARCHY) | 按文档标题层级结构切分 | 有明确章节结构的文档（如技术文档、书籍） |
| **自定义分段** | 1 (CUSTOM) | 手动指定分隔符和切片大小 | 需要精确控制切分逻辑的场景 |

## 二、前端表单配置结构

### 2.1 默认配置对象

```javascript
const DEFAULT_FRAGMENT_DRAFT = {
  strategy: "auto",           // 策略类型: "auto" | "hierarchy" | "custom"
  separator: "\\n\\n",        // 自定义分隔符（仅custom模式）
  chunkSize: 800,             // 切片大小（仅custom模式）
  chunkOverlap: 10,           // 切片重叠字符数（仅custom模式）
  maxLevel: 3,                // 最大层级深度（仅hierarchy模式）
  keepHierarchyInfo: true,    // 保留层级信息（仅hierarchy模式）
  replaceSpaces: true,        // 替换多余空格（custom模式可用）
  removeUrls: false           // 移除URL（custom模式可用）
};
```

### 2.2 文本预处理规则

```javascript
const TEXT_RULES = {
  REPLACE_SPACES: 1,  // 将连续空格替换为单个空格
  REMOVE_URLS: 2      // 移除文本中的URL链接
};
```

## 三、API请求配置结构

### 3.1 策略一：自动分段（推荐）

**前端配置：**
```javascript
{
  strategy: "auto"
}
```

**转换后的API结构：**
```json
{
  "mode": 2
}
```

**特点：**
- 最简单的配置
- 系统自动处理文档结构
- 适用于大部分场景
- 无需额外参数

---

### 3.2 策略二：层级分段

**前端配置：**
```javascript
{
  strategy: "hierarchy",
  maxLevel: 3,              // 识别的最大标题层级（1-6）
  keepHierarchyInfo: true   // 是否在切片中保留层级信息
}
```

**转换后的API结构：**
```json
{
  "mode": 3,
  "hierarchy": {
    "max_level": 3,
    "keep_hierarchy_info": true,
    "text_preprocess_rule": []
  }
}
```

**适用场景：**
- Markdown文档（有 # ## ### 标题）
- Word文档（有标题样式）
- PDF文档（有明确的章节结构）

**效果示例：**
```
文档结构：
# 第一章
## 1.1 小节
内容A
## 1.2 小节
内容B
# 第二章
内容C

切片结果（keep_hierarchy_info=true）：
- 片段1: [第一章 > 1.1 小节] 内容A
- 片段2: [第一章 > 1.2 小节] 内容B
- 片段3: [第二章] 内容C
```

---

### 3.3 策略三：自定义分段

**前端配置：**
```javascript
{
  strategy: "custom",
  separator: "\\n\\n",      // 分隔符（常用: \n\n, \n, 。）
  chunkSize: 800,           // 每个切片的字符数
  chunkOverlap: 10,         // 相邻切片之间的重叠字符数
  replaceSpaces: true,      // 是否替换多余空格
  removeUrls: false         // 是否移除URL
}
```

**转换后的API结构：**
```json
{
  "mode": 1,
  "normal": {
    "text_preprocess_rule": [1],  // [1]=替换空格, [2]=移除URL, [1,2]=两者都启用
    "segment_rule": {
      "separator": "\\n\\n",
      "chunk_size": 800,
      "chunk_overlap": 10
    }
  }
}
```

**参数说明：**

| 参数 | 说明 | 推荐值 | 注意事项 |
|------|------|--------|---------|
| separator | 文本分隔符 | `\\n\\n`（双换行） | 需要转义反斜杠 |
| chunk_size | 切片大小（字符数） | 500-1000 | 太小影响语义完整性，太大影响检索精度 |
| chunk_overlap | 重叠字符数 | 10-50 | 避免切分时丢失关键信息 |
| replaceSpaces | 替换空格 | true | 清理格式问题 |
| removeUrls | 移除URL | false | 仅当URL干扰内容时启用 |

**效果示例：**
```
原文本（以 \n\n 分隔）：
段落1有800字...\n\n段落2有600字...\n\n段落3有500字...

chunkSize=800, chunkOverlap=10 的切片结果：
- 切片1: 段落1（800字）
- 切片2: 段落1末尾10字 + 段落2（610字）
- 切片3: 段落2末尾10字 + 段落3（510字）
```

## 四、配置转换函数

### 4.1 核心转换逻辑

```javascript
function buildFragmentConfig(draft) {
  // 1. 收集文本预处理规则
  const textRules = [];
  if (draft.replaceSpaces) textRules.push(1);  // REPLACE_SPACES
  if (draft.removeUrls) textRules.push(2);     // REMOVE_URLS

  // 2. 根据策略返回不同配置
  if (draft.strategy === "auto") {
    return {
      mode: 2  // FRAGMENT_MODES.AUTO
    };
  }

  if (draft.strategy === "hierarchy") {
    return {
      mode: 3,  // FRAGMENT_MODES.HIERARCHY
      hierarchy: {
        max_level: Number(draft.maxLevel) || 3,
        keep_hierarchy_info: Boolean(draft.keepHierarchyInfo),
        text_preprocess_rule: []  // 层级模式暂不支持预处理规则
      }
    };
  }

  // 默认: custom 模式
  return {
    mode: 1,  // FRAGMENT_MODES.CUSTOM
    normal: {
      text_preprocess_rule: textRules,
      segment_rule: {
        separator: draft.separator,
        chunk_size: Number(draft.chunkSize) || 800,
        chunk_overlap: Number(draft.chunkOverlap) || 10
      }
    }
  };
}
```

### 4.2 反向解析函数

```javascript
function getStrategyFromFragmentConfig(config) {
  const mode = Number(config?.mode || 0);
  if (mode === 3) return "hierarchy";  // FRAGMENT_MODES.HIERARCHY
  if (mode === 1) return "custom";     // FRAGMENT_MODES.CUSTOM
  if (mode === 2) return "auto";       // FRAGMENT_MODES.AUTO
  return "auto";  // 默认
}
```

### 5.1 创建知识库时指定策略

```javascript
// 场景1: 使用自动分段（最简单）
const payload1 = {
  name: "技术文档知识库",
  description: "自动识别结构",
  enabled: true,
  source_type: 1,
  fragment_config: {
    mode: 2  // 自动模式
  }
};

// 场景2: 使用层级分段（适合结构化文档）
const payload2 = {
  name: "产品手册知识库",
  enabled: true,
  source_type: 1,
  fragment_config: {
    mode: 3,
    hierarchy: {
      max_level: 4,              // 识别到 #### 四级标题
      keep_hierarchy_info: true,
      text_preprocess_rule: []
    }
  }
};

// 场景3: 使用自定义分段（精确控制）
const payload3 = {
  name: "API文档知识库",
  enabled: true,
  source_type: 1,
  fragment_config: {
    mode: 1,
    normal: {
      text_preprocess_rule: [1, 2],  // 替换空格 + 移除URL
      segment_rule: {
        separator: "\\n---\\n",      // 使用 --- 作为分隔符
        chunk_size: 1000,            // 较大的切片
        chunk_overlap: 50            // 较大的重叠
      }
    }
  }
};

// 发送请求
await fetch('/api/v1/knowledge-bases', {
  method: 'POST',
  headers: { /* ... */ },
  body: JSON.stringify(payload1)  // 或 payload2, payload3
});
```

## 七、最佳实践建议

### 7.1 策略选择指南

| 文档类型 | 推荐策略 | 配置建议 |
|---------|---------|---------|
| 技术文档、书籍 | hierarchy | max_level=3-4 |
| 产品说明书 | hierarchy | max_level=3, keep_hierarchy_info=true |
| API文档 | custom | separator="\\n---\\n", chunk_size=1000 |
| 通用文本 | auto | 无需配置 |
| 对话记录 | custom | separator="\\n\\n", chunk_size=500 |
| 法律文件 | custom | chunk_size=1200, chunk_overlap=100 |

### 7.2 参数调优建议

**chunk_size（切片大小）：**
- **小文件（<10页）**: 300-500字符
- **中等文件（10-50页）**: 600-800字符
- **大文件（>50页）**: 800-1200字符

**chunk_overlap（重叠大小）：**
- 通常设置为 chunk_size 的 1-10%
- 结构化文档可以较小（10-20）
- 连续性强的文档可以较大（50-100）

**separator（分隔符）：**
- **段落分隔**: `\\n\\n`（推荐）
- **句子分隔**: `。` 或 `. `
- **Markdown分隔**: `\\n---\\n`
- **自定义标记**: 根据文档格式自定义

### 7.3 注意事项

1. **转义问题**: JSON中的反斜杠需要转义，如 `\\n` 表示换行
2. **性能考虑**: chunk_size过小会导致切片数量过多，影响检索性能
3. **语义完整性**: chunk_overlap避免切分时丢失关键信息
4. **文档差异**: 同一知识库的不同文档可以使用不同的切分策略
5. **预览测试**: 正式使用前先用预览接口测试效果

## 八、常见问题

### Q1: 创建知识库后可以修改切分策略吗？

A: 知识库创建后，`fragment_config` 作为默认配置。为已有知识库添加新文档时，可以为该文档指定不同的 `fragment_config`，但已存在的文档不会自动重新切分。如需重新切分，需要调用**重向量化接口**。

### Q2: 如何重新切分已导入的文档？

A: 使用重向量化接口：
```javascript
POST /api/v1/knowledge-bases/{kbCode}/documents/{docCode}/re-vectorized
```
注意：重向量化会根据文档当前关联的 `fragment_config` 重新处理。

### Q3: text_preprocess_rule 在 hierarchy 模式下为什么是空数组？

A: 层级模式主要依赖文档原始结构，暂不支持文本预处理规则，以保持标题层级的准确性。

### Q4: 如何判断当前使用的是哪种策略？

A: 通过 `mode` 值判断：
```javascript
const mode = config.mode;
if (mode === 1) console.log('自定义模式');
if (mode === 2) console.log('自动模式');
if (mode === 3) console.log('层级模式');
```

### Q5: 不同策略的切片质量如何评估？

A: 通过以下方式评估：
1. 使用预览接口查看切片结果
2. 使用相似度检索接口测试检索效果
3. 检查切片的 `metadata.section_title` 等信息是否完整
4. 统计切片数量和平均长度是否合理
