# 文件匹配纠偏机制说明

这份文档专门讲清楚 `app/utils/` 下面这一套“纠偏”机制现在到底是怎么工作的。

目标不是写成设计论文，而是让后来维护这块代码的人一眼能看懂：

1. 现在的原理是什么
2. 各种情况下会发生什么
3. 为什么最后选了这个方案
4. 哪些边界不能乱动

相关代码主要在这几个文件：

- `fuzzy_text_matcher.py`
- `file_path_fuzzy_matcher.py`
- `punctuation_matcher.py`
- `replace_range_resolver.py`

---

## 先说结论

现在这套机制的核心思路是：

**先把两边压成同一种“比较键”去比较，再把命中的位置翻译回原文。**

也就是说：

- 不是直接拿归一化后的字符串去替换
- 也不是用一大堆正则去“猜”各种可能字符
- 而是先比较，再回原文取真实内容

这套做法比“正则双向扩展匹配”更朴素，也更安全。

```text
输入文本 / 路径
        |
        v
原样精确匹配
        |
        +-- 命中 --> 直接返回真实内容（无 warning）
        |
        v
rstrip() / strip() 匹配
        |
        +-- 命中 --> 返回真实内容（有 warning）
        |
        v
双边归一化比较
        |
        +-- 未命中 --------> 失败
        |
        +-- 命中多个候选 --> 失败（歧义，不自动猜）
        |
        +-- 唯一命中 -----> 用索引映射回原文
                                |
                                v
                        返回原文真实子串 / 真实路径（有 warning）
```

---

## 这套机制解决什么问题

主要解决三类模型常见偏差：

1. 中文标点和英文标点混用
   - 例如：`：` 和 `:`
   - 例如：`，` 和 `,`

2. 特殊空白字符
   - 例如：全角空格 `U+3000`
   - 例如：非断行空格 `U+00A0`

3. 中文和英文/数字之间误插空格
   - 例如：`topic 主题`
   - 实际可能是：`topic主题`

注意：

这套机制**不是**为了做“强力模糊搜索”。  
它只处理这些非常窄、非常高频、非常确定的偏差。

---

## 适用范围

### 路径纠偏

会用到的地方：

- `read_file`
- `read_files`
- `WorkspaceTool.resolve_path_fuzzy()`

### 文本锚点纠偏

会用到的地方：

- `edit_file`
- `multi_edit_file`
- `edit_file_range`
- `multi_edit_file_range`

### 不适用

这套机制**不**用于：

- `write_file`
- shell / command 路径解析
- 编辑距离
- 模糊打分
- 跨目录猜文件

---

## 现在的匹配顺序

无论是文本锚点还是区间锚点，顺序都尽量保持简单：

1. 原样精确匹配
2. `rstrip()` 后匹配
3. `strip()` 后匹配
4. 双边归一化后匹配

只要前一档命中，就不会进入下一档。

这点很重要：  
**纠偏永远是 fallback，不是默认行为。**

---

## “双边归一化”到底是什么意思

就是把：

- 用户输入
- 文件原文

都变成同一种“比较键”，再比较。

当前会做的归一化有三类：

### 1. 中文标点转英文标点

例如：

- `，` -> `,`
- `。` -> `.`
- `：` -> `:`
- `；` -> `;`
- `！` -> `!`
- `？` -> `?`
- `（` -> `(`
- `）` -> `)`
- `《` -> `<`
- `》` -> `>`
- `【` -> `[`
- `】` -> `]`
- `、` -> `,`
- `—` / `－` -> `-`

### 2. 特殊空白转普通空格

例如：

- `U+3000` 全角空格
- `U+00A0` 非断行空格
- 其他窄空格 / 宽空格

都会统一变成普通空格。

### 3. 只删除中英文边界空格

只删这种情况：

- 中文 + 空格 + 英文/数字
- 英文/数字 + 空格 + 中文

例如：

- `topic 主题` -> `topic主题`
- `第 1 章` -> `第1章`

注意：

- 不会全量去空格
- 不会改英文单词内部空格
- 不会改纯中文词组内部空格

---

## 为什么还需要“索引映射”

因为归一化以后，字符串长度可能会变。

例如：

- 原文：`topic 主题`
- 归一化后：`topic主题`

如果只保留归一化后的文本，就会丢失一个关键事实：

**归一化后命中的区间，在原文里到底对应哪一段？**

所以 `normalize_with_index_map()` 做了两件事：

1. 生成归一化后的文本
2. 记录归一化后每个字符，对应原文哪个索引

这样一来：

- 先在归一化后的文本里找到命中位置
- 再用索引映射回原文
- 取出真实子串

所以最终替换、展示、warning 里用的，都是**原文真实内容**。

---

## 路径场景：它到底怎么工作

路径场景很简单，不做复杂猜测。

逻辑是：

1. 先看原路径是否存在
2. 如果不存在，只在**当前父目录**下枚举文件
3. 把输入文件名和目录里的文件名都归一化
4. 如果归一化后只有一个候选相等，就命中它
5. 如果没有候选，或者候选不止一个，就失败

### 路径例子 1：精确命中

输入：

```text
AgentLoop完整文档.md
```

目录里：

```text
AgentLoop完整文档.md
```

结果：

- 直接命中
- 没有 warning

### 路径例子 2：中文英文边界空格

输入：

```text
AgentLoop 完整文档.md
```

目录里：

```text
AgentLoop完整文档.md
```

结果：

- 自动命中 `AgentLoop完整文档.md`
- 会有 warning

### 路径例子 3：中英文标点

输入：

```text
MCP: 文档.md
```

目录里：

```text
MCP：文档.md
```

结果：

- 自动命中 `MCP：文档.md`
- 会有 warning

### 路径例子 4：多候选

输入：

```text
topic 主题.md
```

目录里同时有：

```text
topic主题.md
topic：主题.md
```

结果：

- 不自动猜
- 直接失败

## 文本锚点场景：它到底怎么工作

这个场景对应：

- `edit_file`
- `multi_edit_file`

流程如下：

1. 先尝试原样匹配 `old_string`
2. 失败后试 `rstrip()`
3. 再失败试 `strip()`
4. 再失败才走双边归一化
5. 归一化命中后，用索引映射回原文，拿真实子串
6. 如果归一化后有多个候选，不自动修

### 文本例子 1：精确命中

输入：

```text
old_string = "hello, world"
```

文件里：

```text
hello, world
```

结果：

- 直接命中
- 没有 warning

### 文本例子 2：`rstrip()` 命中

输入：

```text
old_string = "hello, world   "
```

文件里：

```text
hello, world
```

结果：

- 自动命中 `hello, world`
- 会有 warning

### 文本例子 3：`strip()` 命中

输入：

```text
old_string = "\n  hello, world  \n"
```

文件里：

```text
hello, world
```

结果：

- 自动命中 `hello, world`
- 会有 warning

### 文本例子 4：中文英文标点

输入：

```text
old_string = "MCP: 文档"
```

文件里：

```text
MCP：文档
```

结果：

- 归一化后命中
- 最终返回的真实子串是：`MCP：文档`
- 会有 warning

### 文本例子 5：边界空格

输入：

```text
old_string = "topic 主题"
```

文件里：

```text
topic主题
```

结果：

- 归一化后命中
- 返回真实子串：`topic主题`
- 会有 warning

### 文本例子 6：反过来

输入：

```text
old_string = "topic主题"
```

文件里：

```text
topic 主题
```

结果：

- 归一化后命中
- 返回真实子串：`topic 主题`
- 会有 warning

### 文本例子 7：归一化后多候选

文件里：

```text
topic：主题
...
topic:主题
```

输入：

```text
old_string = "topic:主题"
```

结果：

- 归一化后会发现候选不唯一
- 不自动纠偏
- 返回诊断信息，要求重新读文件并复制精确文本

## 区间锚点场景：它怎么工作

这个场景对应：

- `edit_file_range`
- `multi_edit_file_range`

这里比普通 `edit_file` 更严格。

对于 `replace_start` / `replace_end`：

1. 先精确找
2. 失败后做归一化匹配
3. 如果归一化后不是唯一候选，直接报歧义
4. 只有唯一候选才继续算区间

### 区间例子 1：唯一纠偏

输入：

```text
replace_start = "## MCP: 文档"
replace_end = "## 下一节"
```

文件里：

```text
## MCP：文档
...
## 下一节
```

结果：

- `replace_start` 自动纠偏到 `## MCP：文档`
- 区间正常解析
- 会有 warning

### 区间例子 2：归一化后歧义

文件里：

```text
## MCP：文档
...
## MCP:文档
```

输入：

```text
replace_start = "## MCP: 文档"
```

结果：

- 归一化后候选不唯一
- 直接报歧义
- 不自动选第一个

---

## Warning 规则

只有一种情况没有 warning：

- 原样精确命中

只要发生了以下任一种纠偏，就应该有 warning：

- `rstrip()` 命中
- `strip()` 命中
- 归一化后命中
- 路径纠偏命中
- 区间锚点纠偏命中

warning 的意义不是报错，而是告诉模型：

- 这次系统替你纠偏了
- 下次请直接使用真实文本或真实路径

---

## 为什么最后选了这套方案

中间其实考虑过更“聪明”的方案，比如：

- 正则双向扩展匹配
- 更宽的模糊规则

最后没有选，原因很简单：

1. 太聪明的方案更难解释
2. 匹配面一宽，就更容易误命中
3. 后续维护的人更难判断它到底为什么命中

现在这套方案虽然没有那么“炫”，但优点很明确：

- 逻辑简单
- 行为可预测
- 命中结果可解释
- 出错时更容易排查

这比“尽量多帮模型猜对”更重要。

---

## 后续修改这套机制时，必须自检

如果以后要改这里，先问自己这几个问题：

1. 有没有破坏“精确优先”
2. 有没有破坏“唯一候选才纠偏”
3. 有没有把归一化结果直接拿去替换真实内容
4. 有没有把路径纠偏做成跨目录猜测
5. 有没有把人类侧 detail 和 AI 侧 content 混在一起
6. 有没有把简单比较逻辑演变成重型 fuzzy 机制

如果其中任何一条答案是“有可能”，那就应该先写方案，再改代码。
